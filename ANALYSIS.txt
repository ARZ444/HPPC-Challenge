Explanation of my optimization strategy:
I started from my working Part 1 MPI program and kept the same row decomposition, the MPI_Allreduce(MPI_MAX) convergence check and the MPI_Wtime timing. The optimization I tried is to overlap the halo (ghost row) exchange with the computation. In Part 1 I used blocking MPI_Send and MPI_Recv, so each process had to finish sending and receiving its two ghost rows before it could start the Jacobi update. In the optimized version I changed this to the non-blocking calls MPI_Irecv and MPI_Isend (Module 6a): each process posts the receives for its top and bottom ghost rows and the sends of its first and last real rows, and then carries on instead of waiting.

While those messages are still being sent, the process updates the interior rows that do not need any ghost cell. Only the first and last real rows of each block actually need the received halo, so all the rows in between can be done right away. After that I call MPI_Waitall to make sure the ghost rows have arrived, and only then update the first and last rows. This is the "overlap communication with computation" idea from the notes (Module 9) - the time spent moving the halo is hidden behind useful work instead of just waiting.

I only changed how the communication is scheduled, not the maths. Every interior point is still computed from the old grid only, the local maximum is taken over the same interior points, and the global maximum still comes from MPI_Allreduce(MPI_MAX). So the optimized version gives exactly the same output as the serial and Part 1 versions for np = 1, 2, 4 and 8: 10001 iterations and a final maximum change of 1.273176e-02.

What worked well:
The most important thing is that correctness stayed exact - the non-blocking version still matches the serial baseline (10001 iterations and 1.273176e-02) for 1, 2, 4 and 8 processes, so splitting the update into halo-independent and halo-dependent rows did not change the result. The non-blocking version is also the proper, scalable communication pattern: on a real cluster where the halos cross a network and the runtime can move messages in the background, hiding that latency behind the interior computation is exactly where this would help.

What challenges did you face:
The main problem was that the optimization did not actually run faster on my test machine, which only has 2 physical cores. At np = 2 (3 runs each) the blocking Part 1 version averaged about 4.4 seconds, while the non-blocking version averaged about 6.0 seconds, so it was slower. The reason is that real overlap needs spare hardware to move the messages while the cores are computing; with both cores fully busy and no background progress, the non-blocking messages only really move during MPI_Waitall, so there is no overlap. The halos are also tiny (one row of 1000 doubles, about 8 KB) and go through shared memory between processes on the same machine, so the communication is already very cheap compared with the heavy per-iteration work over a million points - there is almost nothing to hide. The extra cost of splitting the update into three phases then shows up as a small slowdown. The other tricky parts were making sure the deferred first and last rows still read only the old grid (so the answer stays the same) and getting the boundary rows classified correctly at each rank. Since the assessment notes say a low speedup is fine if it is explained, my conclusion is that this optimization is correct and is the right design for a latency-bound cluster, but it cannot show a benefit on a single 2-core shared-memory machine.
